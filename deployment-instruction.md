# Workstation Deployment Instructions

This runbook is for an agent deploying the single-Team Team Memory service and
Human Portal on one persistent workstation. Follow the stop conditions and do
not improvise around identity or secret recovery.

## Direct answer: do we need to retrieve an Admin Key?

No Admin Key is generated by the service, and there is no API that can retrieve
one.

For a new Human Portal deployment, leave `TEAM_MEMORY_ADMIN_API_KEY` empty. The
operator supplies a one-time bootstrap secret, the first user signs in through
OIDC, and that user claims the first `Owner` membership. Daily administration
then uses the Owner's Human Session, not a shared Admin Key.

`TEAM_MEMORY_ADMIN_API_KEY` exists only for the legacy headless enrollment flow
or a controlled migration from that flow. It is an operator-provided environment
secret, not a server-issued secret. If it is configured alongside Human
Identity, it stops authenticating as soon as bootstrap is claimed. It is not a
break-glass or Owner-recovery credential.

Keep these secret classes distinct:

| Secret | Who creates it | Returned by server | Operational rule |
| --- | --- | --- | --- |
| Bootstrap secret | Deployment operator | No | Needed to claim the first Owner; retain in deployment configuration because the current identity configuration requires it, but it is permanently inactive after claim. |
| Static Admin Key | Deployment operator | No | Optional legacy/migration input; omit for a new Portal deployment. |
| Secret pepper | Deployment operator | No | Treat as installation identity. Losing or changing it invalidates stored secret digests. Back it up securely. |
| OIDC client secret | OIDC provider | No | Store with deployment secrets and rotate through the provider. |
| OIDC flow secret | Deployment operator | No | Protects transient login-flow state; rotation interrupts in-flight logins. |
| Enrollment token | Portal/API | Yes, once | Deliver only to the target Agent. It normally expires after 15 minutes. |
| Agent API key | Enrollment exchange | Yes, once | Store on the Agent workstation. If lost, revoke it and issue a new Enrollment; it cannot be retrieved. |

## Current delivery boundary

The repository currently provides:

- the Team Memory backend, PostgreSQL migrations, Agent Registry, Human
  Identity APIs, and Capsule Channel HTTP APIs;
- a React/TypeScript Portal under `web/`;
- a Compose topology for PostgreSQL, Team Memory, and the local Qwen embedding
  server;
- `deploy/workstation/compose.yaml`, which binds PostgreSQL and the backend to
  loopback, requires a deployment-specific PostgreSQL password, and adds a
  Caddy Portal gateway;
- `deploy/workstation/Caddyfile`, which serves `web/dist`, proxies the public
  API, and issues an internal-CA TLS certificate.

The backend container does not serve `web/dist`. Production deployment must
serve the Portal and proxy `/v1/*` to the backend from the same origin.

Do not declare paxl Capsule delivery complete only because the backend is
healthy. A deployment requiring paxl must pass the CLI capability gate in the
paxl section below; do not infer support from the Team Memory backend version.

## Required inputs

Collect these before changing the workstation:

- repository checkout at the version to deploy;
- Docker Engine or Docker Desktop with Compose v2.24.4 or newer. The checked-in
  workstation override uses the Compose `!override` tag;
- Node.js 18+ and npm for building the Portal;
- a stable workstation hostname, DNS record, and TLS termination for a
  persistent multi-user deployment;
- a confidential OIDC client whose issuer supports discovery and whose allowed
  scopes include `openid`, `profile`, and `email`;
- an OpenAI-compatible extraction provider and API key, or an explicitly
  accepted non-production `noop` extraction mode;
- a secure location for deployment secrets and PostgreSQL backups;
- enough disk space for PostgreSQL and the Qwen model cache.

STOP if the deployment has no OIDC provider. The current Human Plane does not
implement local passwords, passkeys, or magic links.

STOP if the workstation already has a Team Memory Compose volume that is not
understood. Inspect and back it up before running any cleanup command. Never use
`docker compose down -v` on a persistent installation.

## 1. Choose the URL shape

### Persistent deployment

Use one HTTPS origin, for example:

```text
Portal:       https://memory.example.internal/
API:          https://memory.example.internal/v1/...
Health:       https://memory.example.internal/healthz
OIDC callback:https://memory.example.internal/v1/auth/callback
```

Expose only the TLS gateway to the network. Always include
`deploy/workstation/compose.yaml`; it replaces the checked-in development port
bindings with `127.0.0.1` bindings for PostgreSQL port `55432` and backend port
`58080`. Verify the rendered bindings before startup. Do not run persistent
deployment commands with only the base `compose.yaml`.

### Local acceptance only

If the OIDC provider permits a localhost HTTP callback, the Vite development
server can proxy the API:

```text
Portal:       http://localhost:5173/
OIDC callback:http://localhost:5173/v1/auth/callback
Backend:      http://localhost:58080/
```

Set `TEAM_MEMORY_HUMAN_COOKIE_SECURE=false` only for this plain-HTTP local
acceptance mode. Leaving it `true` over HTTP causes a login loop because the
browser drops the Secure cookies.

## 2. Create and store deployment secrets

Choose one explicit bootstrap custody ceremony before generating secrets:

- Preferred: the operator generates the bootstrap secret and places it in the
  workstation secret store before handing the deployment to the Agent.
- Delegated: the deployment Agent generates it, deposits it directly into an
  operator-accessible secret store, and returns only the secret reference. The
  Agent must not put the value in chat, logs, or the handoff report.

The first Owner must be able to access the bootstrap secret through that secure
channel until bootstrap claim succeeds. It is the secret needed for first-time
setup; there is no Admin Key to retrieve.

Generate independent secrets. Do not reuse one value across fields.

```sh
openssl rand -hex 32
openssl rand -hex 32
openssl rand -hex 32
```

Use the outputs for the bootstrap secret, secret pepper, and OIDC flow secret.
Generate one additional hex value for `TEAM_MEMORY_POSTGRES_PASSWORD`. Store
them in the workstation secret manager or in a root/operator-readable `.env`
file with mode `0600`. Never commit `.env`.

For a recommended persistent Portal deployment, configure at least:

```dotenv
TEAM_MEMORY_POSTGRES_PASSWORD=<independent-uri-safe-hex-secret>
TEAM_MEMORY_PORTAL_HOST=memory.example.internal
TEAM_MEMORY_ADMIN_API_KEY=
TEAM_MEMORY_BOOTSTRAP_SECRET=<independent-high-entropy-secret>
TEAM_MEMORY_SECRET_PEPPER=<independent-high-entropy-secret-at-least-32-characters>
TEAM_MEMORY_OIDC_ISSUER=https://<oidc-provider-issuer>
TEAM_MEMORY_OIDC_CLIENT_ID=<client-id>
TEAM_MEMORY_OIDC_CLIENT_SECRET=<client-secret>
TEAM_MEMORY_OIDC_REDIRECT_URL=https://memory.example.internal/v1/auth/callback
TEAM_MEMORY_OIDC_FLOW_SECRET=<independent-high-entropy-secret>
TEAM_MEMORY_PORTAL_URL=https://memory.example.internal/
TEAM_MEMORY_HUMAN_COOKIE_SECURE=true
TEAM_MEMORY_MEMBER_GRANTABLE_PERMISSIONS=observe,search,get,channel_send,channel_receive

TEAM_MEMORY_EXTRACTOR_MODE=openai
TEAM_MEMORY_EXTRACTOR_BASE_URL=https://api.deepseek.com
TEAM_MEMORY_EXTRACTOR_API_KEY=<provider-api-key>
TEAM_MEMORY_EXTRACTOR_MODEL=deepseek-v4-flash
TEAM_MEMORY_EXTRACTION_VERSION=v1
TEAM_MEMORY_WIKI_HINT_ENABLED=false
```

Do not set `TEAM_MEMORY_API_KEYS` in this mode. Compatibility API keys and the
on-prem identity settings are mutually exclusive.

Validate only the Compose syntax and interpolation without printing resolved
secrets:

```sh
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  config --quiet
```

This does not validate Team Memory's authentication invariants. The startup and
bounded log checks in step 4 are the application-configuration validation.

## 3. Configure the OIDC client

At the OIDC provider:

1. Create a confidential web client.
2. Register the exact `TEAM_MEMORY_OIDC_REDIRECT_URL`.
3. Allow the `openid`, `profile`, and `email` scopes.
4. Ensure ID tokens contain stable issuer/subject values and the `email`,
   `email_verified`, and `name` claims.
5. Prefer an invite-restricted provider audience or group. Team Memory itself
   remains invite-only after login, but provider restriction reduces exposure.

The redirect URL must match exactly, including scheme, hostname, port, and
path. Do not put a reverse-proxy-only hostname in the environment if the
browser reaches a different hostname.

## 4. Start the backend

From the repository root:

```sh
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  up -d --build --wait postgres qwen-embedding team-memory
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  ps
curl -fsS http://127.0.0.1:58080/healthz
```

The application applies PostgreSQL migrations during startup. Do not run a
separate migration binary.

If startup fails, inspect bounded logs without copying secrets into a ticket:

```sh
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  logs --tail=200 postgres team-memory qwen-embedding
```

Expected result: PostgreSQL and Qwen are healthy, Team Memory stays running,
`/healthz` succeeds, and the Team Memory logs contain no configuration or
migration failure. An immediate Team Memory exit is a failed configuration
check even when `docker compose config --quiet` succeeded.

## 5. Build and serve the Portal

Build the checked-in Portal:

```sh
cd web
npm ci
npm test
npm run build
```

The checked-in workstation Caddy service serves `web/dist/` from the same
origin as the API. Validate and start it:

```sh
cd ..
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  run --rm --no-deps caddy \
  caddy validate --config /etc/caddy/Caddyfile --adapter caddyfile
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  up -d caddy
curl -kfsS https://memory.example.internal/healthz
```

The checked-in Caddy configuration:

- serves the SPA and falls back to `index.html` for client routes;
- proxies `/v1/*` and `/healthz` to `team-memory:8080` on the private Compose
  network;
- preserves `Host`, `X-Forwarded-Proto`, cookies, and redirects;
- terminates TLS for a persistent deployment;
- does not enable credentialed wildcard CORS.

It uses Caddy's internal CA so the path is deterministic on a private
workstation. `curl -k` is only a pre-trust smoke check. Before browser/OIDC use,
the operator must export the root certificate from the persistent Caddy data
volume and explicitly approve installing it in the workstation and client trust
stores. Do not silently change system trust from an Agent run. For an existing
organizational TLS gateway, replace the Caddy service with that approved gateway
and validate the same route/cookie behavior.

Export and identify the public root certificate for the operator without
installing it:

```sh
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  cp caddy:/data/caddy/pki/authorities/local/root.crt \
  /tmp/team-memory-caddy-root.crt
openssl x509 -in /tmp/team-memory-caddy-root.crt \
  -noout -subject -issuer -fingerprint -sha256
```

STOP and ask the operator to approve the certificate and trust-store change.
After approval, the operator or an authorized device-management system installs
the certificate on every browser workstation. Then rerun the external health
check without `-k` before starting OIDC login.

For local acceptance instead, run `npm run dev` in `web/`. It proxies `/v1` to
`http://localhost:58080` by default.

## 6. Claim the first Owner

1. Open the Portal URL.
2. Start OIDC login and complete the provider flow.
3. Confirm the Portal shows the bootstrap state, not `501 Not Configured`.
4. Retrieve the bootstrap secret through the custody ceremony from step 2 and
   enter it in the bootstrap form. Do not copy it through chat or a ticket.
5. Claim the first Owner.
6. Confirm `/v1/me` now reports an active Owner membership.

Only one concurrent bootstrap claim can succeed. After success:

- bootstrap is permanently closed in PostgreSQL;
- the static Admin Key, if present for migration, stops authenticating;
- the bootstrap secret is not a recovery credential;
- normal administration uses the Owner's OIDC-backed Human Session.

Before inviting ordinary members, invite a second trusted person, have that
person accept the invitation, and promote the active membership to Owner. The
service prevents removal of the final active Owner, but it cannot recover an
installation when the only Owner permanently loses access to the OIDC identity.

## 7. Register and enroll Agent clients

For each Agent:

1. An active Human member (`Owner`, `Admin`, or `Member`) creates an Agent
   Profile with a unique `agent_id`.
2. The Agent owner creates a short-lived Enrollment with a label and explicit
   permissions.
3. Include `channel_send` and/or `channel_receive` for Capsule transport.
4. Copy the one-time Enrollment token directly to the target Agent workstation.

Choose exactly one consumer for that token:

- Non-paxl client: the Agent calls `POST /v1/agent-enrollments/exchange` once
  and stores the returned `tm_key_...` in its credential store.
- paxl client: do not exchange the token manually. Continue to section 8 with a
  fresh pending token; `paxl channel connect` owns the single exchange and
  credential storage.

After the selected client succeeds, confirm the Portal shows the Enrollment as
consumed and the Credential as active. Do not paste the API key back into the
Portal.

Minimum public-API verification for a non-paxl Agent, run only where its
credential is already available:

```sh
curl -fsS \
  -H 'Authorization: Bearer <tm_key_...>' \
  https://memory.example.internal/v1/agent-identity
```

If an Enrollment token is lost, expired, consumed, or revoked, issue a new
Enrollment. If an Agent API key is lost, revoke its Credential and issue a new
Enrollment. Neither secret is retrievable.

## 8. paxl Capsule Channel gate

Before configuring paxl, verify the installed binary rather than relying on an
ADR example:

```sh
paxl version
paxl channel connect --help
```

STOP if the second command reports that `channel` is not a command. The
installed paxl build does not contain the accepted on-prem channel surface.
Obtain and verify a build containing that surface before continuing.

For a build that implements the accepted CLI contract, use its actual `--help`
output as the source of truth. Supply a fresh pending Enrollment token that has
not been manually exchanged; the intended flow is:

```sh
paxl channel connect onprem \
  --url https://memory.example.internal \
  --enrollment-token <tm_enroll_...>

paxl inbox list --channel onprem
```

Do not retry `channel connect` with a token that may already have been consumed.
Check the Portal Enrollment state and create a new Enrollment when necessary.

Then send a small test Capsule to another active, routable Agent and verify
send, inbox materialization, accept, and archive. Do not mark the deployment
complete until the receiver has materialized the Capsule locally and the
remote envelope is accepted.

## 9. Acceptance checklist

Run and record non-secret evidence for all applicable checks:

- the two-file `docker compose ... config --quiet` command succeeds;
- `docker compose ps` reports healthy dependencies, a running backend, and
  PostgreSQL/backend published addresses on `127.0.0.1` only;
- the internal and external `/healthz` URLs succeed.
- the Caddy configuration validates and the operator has approved the required
  TLS trust step.
- `/v1/auth/login` redirects to the expected OIDC provider.
- OIDC callback returns to the exact Portal URL without a cookie loop.
- exactly one user claims bootstrap and becomes Owner.
- a second active Owner exists before production handoff.
- an invitation can be accepted by a different OIDC identity.
- an Agent can be created, enrolled, exchanged, and authenticated.
- the Agent Directory list/get endpoints show only routable Agents as allowed.
- an Agent with channel permissions can exercise the Capsule HTTP flow.
- `make onprem-e2e` passes from a clean, isolated test project when Docker image
  registries are reachable.
- if paxl is in scope, its on-prem channel CLI gate and two-Agent Capsule flow
  pass.

Never include bootstrap secrets, OIDC secrets, Enrollment tokens, Agent keys,
cookies, Authorization headers, or `.env` contents in the deployment report.

## 10. Backup, upgrade, and recovery

Create a PostgreSQL backup before every upgrade:

```sh
mkdir -p backups
chmod 700 backups
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  exec -T postgres \
  pg_dump -U team_memory -d team_memory -Fc \
  > "backups/team-memory-$(date -u +%Y%m%dT%H%M%SZ).dump"
```

Back up the secret pepper and OIDC configuration separately from the database.
A database backup without the matching pepper cannot recover existing Sessions,
Invitations, Enrollments, or Agent credentials.

Upgrade sequence:

```sh
git pull --ff-only
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  build team-memory
docker compose \
  -f compose.yaml \
  -f deploy/workstation/compose.yaml \
  up -d --wait
cd web
npm ci
npm test
npm run build
```

After the reverse proxy serves the new `web/dist`, repeat health, login, Agent
identity, and Capsule smoke checks.

Do not blindly roll back the application after a migration. Restore a matched
application/database backup pair when a migration is not backward compatible.

Recovery boundaries:

- Lost Agent key: revoke and re-enroll the Agent.
- Lost Enrollment or Invitation token: revoke if still pending and issue a new
  one.
- Lost secret pepper: restore it from the deployment secret backup; there is no
  API recovery path.
- Lost sole Owner identity: the static Admin Key and bootstrap secret cannot
  recover ownership. Restore OIDC access or recover from a previously tested
  operational backup/process.
- Lost PostgreSQL volume: restore the database backup and the matching secrets.

## 11. Agent handoff record

The deployment Agent should hand back only:

- deployed Git commit or release;
- Portal/API origin and OIDC issuer, but not client secret;
- bootstrap secret-store reference and claimant, but never the secret value;
- Compose service health and external health-check result;
- bootstrap claimed status and Owner count, but not identities unless requested;
- backup location and restore-test date;
- Agent IDs, permission sets, and credential IDs, never secret values;
- paxl version and whether the on-prem channel CLI/flow passed;
- known blockers and exact failing non-secret command/output.
